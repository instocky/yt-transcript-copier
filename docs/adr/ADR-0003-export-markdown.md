# **ADR-0003 — Экспорт транскрипта YouTube в Markdown файл**

## Status

Принято

---

## Контекст

Текущее расширение Chrome извлекает транскрипт YouTube и копирует его в буфер обмена через контекстное меню.

Текущие возможности:

- устойчивое извлечение транскрипта из разных DOM-структур YouTube
- автоматическое открытие панели транскрипта
- удаление таймкодов и нормализация текста
- добавление метаданных (title + URL)

Ограничения:

- результат доступен только через clipboard
- отсутствует персистентное хранение
- нет структурированного формата для дальнейшего использования (RAG, поиск, база знаний)

---

## Решение

Расширить pipeline и добавить экспорт транскрипта в файл **Markdown** через `chrome.downloads.download`.

---

## Архитектура

```text
extract → normalize → enrich → render(markdown) → download(file)
```

---

## Дизайн решения

### 1. Формат выходного файла

Markdown с YAML frontmatter:

```md
---
title: '{title}'
source: 'youtube'
author: '{channel}'
url: '{url}'
date: '{date}'
video_id: '{id}'
---

# {title}

## Summary

{empty}

## Transcript

{chunks}
```

Особенности:

- `Summary` всегда присутствует (даже если пустой)
- все значения в YAML — строки
- дата хранится как строка (`YYYY-MM-DD`)

---

### 2. Стратегия именования файла

Шаблон:

```text
yt_{author}_{slug}_{yyyy-mm-dd}.md
```

---

### 3. Нормализация

#### author (channel)

- lowercase
- пробелы → `-`
- обрезка до 50 символов

---

#### slug (из title)

- lowercase
- удаление символов вне `[a-z0-9 -]`
- пробелы → `-`
- схлопывание повторных `-`
- обрезка до 80 символов

---

#### date

- формат: `YYYY-MM-DD`

---

### 4. Fallback-значения

- нет автора → `unknown`
- нет title → `video-{id}`

---

### 5. Обработка транскрипта

Вход:

- уже очищенный текст без таймкодов
- разбит на абзацы (как в текущей реализации)

Обработка:

- объединение сегментов
- формирование абзацев (~500–1000 символов)

Выход:

```md
paragraph 1

paragraph 2
```

---

### 6. Механизм сохранения

Используется:

```text
chrome.downloads.download
```

Параметры:

- `filename` — сгенерированное имя
- `conflictAction` — `uniquify`
- MIME — `text/markdown`

Поведение:

- файл сохраняется в папку загрузок браузера
- без дополнительного диалога

---

### 7. Внутренний data contract

```json
{
  "title": "...",
  "channel": "...",
  "url": "...",
  "video_id": "...",
  "date": "...",
  "transcript": "..."
}
```

Является единым источником данных для:

- генерации имени файла
- генерации markdown

---

### 8. UX изменения

Контекстное меню:

```text
📋 Скопировать транскрипт
⬇️ Сохранить как Markdown
```

Обратная связь:

- уведомление об успешном сохранении (с именем файла)
- сообщение об ошибке (если нет транскрипта)

---

## Последствия

### Плюсы

- появляется персистентное хранение
- стандартизированный формат (совместим с Obsidian / Notion / RAG)
- минимальные изменения текущей архитектуры
- повторное использование существующего extraction pipeline

---

### Минусы

- нет полного контроля над путём сохранения (ограничение браузера)
- возможны конфликты имён (решается через `uniquify`)
- зависимость от Chrome Downloads API

---

## Рассмотренные альтернативы

### File System Access API

Отклонено:

- требует явного user interaction
- ухудшает UX для быстрого сохранения

---

### Локальный backend / агент

Отклонено на текущем этапе:

- увеличивает сложность
- не требуется для MVP

---

## Дальнейшее развитие

- добавление таймкодов в транскрипт
- автоматическое summary
- тегирование и категоризация
- экспорт в knowledge base
- переход на локальный агент для контроля файловой системы

---

## Итог

Решение добавляет слой экспорта без изменения существующей логики извлечения, фиксирует формат данных и создаёт основу для дальнейшего развития (поиск, RAG, knowledge base).
