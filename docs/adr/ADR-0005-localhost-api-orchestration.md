# ADR-0005 - Локальный localhost API для пакетной оркестрации экспорта транскриптов

## Status

Предлагается

---

## Контекст

Текущее расширение ориентировано на интерактивную работу внутри Chrome:

- пользователь открывает страницу видео YouTube
- пользователь вызывает действие через контекстное меню
- расширение инжектит `content.js`
- логика извлечения выполняется в контексте активной вкладки
- результат либо копируется в буфер обмена, либо сохраняется как Markdown

Такой сценарий хорошо работает для ручного использования, но не предоставляет программной точки входа для небольшого ежедневного batch-процесса, запускаемого из Python.

Целевой сценарий:

- у пользователя уже открыт Chrome с установленным расширением
- Python получает список YouTube URL
- ролики должны обрабатываться последовательно
- не должна использоваться browser automation / UI-эмуляция
- нагрузка небольшая: порядка 20-30 роликов в сутки

Ограничения и вводные:

- в расширении уже существует рабочий pipeline извлечения и экспорта
- желательно переиспользовать текущую логику, а не дублировать парсинг транскрипта в Python
- допустим короткоживущий локальный процесс, который поднимается на время задания и завершается после обработки очереди

---

## Решение

Добавить в архитектуру легковесный localhost API между Python и Chrome extension.

Расширение становится browser-side оркестратором:

- получает задания с локального HTTP endpoint на `127.0.0.1`
- последовательно открывает и обрабатывает YouTube-вкладки
- вызывает существующие функции извлечения и экспорта внутри вкладки
- отправляет обратно статус обработки

Python становится run-scoped координатором:

- поднимает короткоживущий локальный HTTP сервер
- публикует очередь URL
- принимает статусы выполнения
- завершает работу после исчерпания очереди

Это решение выбрано вместо UI-эмуляции и вместо Chrome Native Messaging.

Дополнительно в решение включается минимальный reliability-слой:

- `batch_id` для группировки одного запуска
- явные `job_id` для каждого элемента очереди
- state machine для статусов обработки
- идемпотентная отчетность со стороны расширения
- хранение рабочего состояния в `chrome.storage`
- healthcheck между расширением и локальным Python-процессом
- dual-mode результата: инициирование download и возврат структурированного результата в API
- reclaim просроченного lease
- ограниченные retry с backoff
- execution timeout для одного job
- single-flight guard на стороне расширения

---

## Архитектура

```text
Python batch runner
  -> поднимает localhost API на 127.0.0.1
  -> отдает очередь URL и принимает статусы

Chrome extension background/service worker
  -> опрашивает localhost API
  -> открывает следующую YouTube-вкладку
  -> инжектит content.js
  -> вызывает extractAndSaveMarkdown()
  -> отправляет результат обратно в localhost API

content.js
  -> переиспользует существующий extraction pipeline
  -> при необходимости автоматически открывает transcript panel
  -> формирует markdown
  -> инициирует скачивание файла
```

## Дизайн решения

### 1. Разделение ответственности

Python отвечает за:

- запуск и завершение batch-задачи
- определение очереди URL
- агрегацию статусов
- логирование и итоговую сводку

Расширение отвечает за:

- управление вкладками Chrome
- последовательное выполнение заданий
- запуск extraction внутри страницы
- инициирование скачивания Markdown-файла

---

### 2. Модель коммуникации

В качестве моста используется HTTP по `127.0.0.1`.

Принципы выбора:

- без постоянного внешнего сервиса
- без зависимости от Selenium / Playwright
- с простым request/response контрактом
- с удобной локальной отладкой

`127.0.0.1` предпочтительнее, чем `localhost`, чтобы избежать лишних вопросов с разрешением имени хоста.

На текущем этапе допустим polling как MVP-подход, но контракт должен позволять в дальнейшем перейти на long polling или push-модель без пересмотра общей архитектуры.

---

### 3. Минимальный API-контракт

Базовые endpoint'ы:

- `POST /jobs`
  - регистрирует новый batch
  - payload: `batch_id` и список YouTube URL

- `GET /health`
  - возвращает состояние локального Python-процесса
  - используется расширением для проверки доступности orchestration-layer

- `GET /next`
  - возвращает следующее задание для расширения
  - операция должна быть атомарной
  - задание выдается вместе с lease/attempt metadata

- `POST /report`
  - принимает статус выполнения
  - payload включает `job_id`, URL, attempt, состояние, при наличии имя файла, структурированный результат и текст ошибки

Пример payload для статуса:

```json
{
  "batch_id": "batch-2026-04-10-001",
  "job_id": "job-0001",
  "url": "https://www.youtube.com/watch?v=...",
  "attempt": 1,
  "status": "done",
  "filename": "yt_channel_title_2026-04-10.md",
  "result": {
    "title": "Video title",
    "video_id": "abc123",
    "markdown": "..."
  },
  "error": null
}
```

Пример ответа `/next`:

```json
{
  "batch_id": "batch-2026-04-10-001",
  "job_id": "job-0001",
  "url": "https://www.youtube.com/watch?v=...",
  "attempt": 1,
  "lease_ttl": 30
}
```

Состояние `done` вычисляется на стороне Python как производное от статусов всех элементов очереди, поэтому отдельный endpoint `/done` не является обязательным.

Минимальная retry-политика:

- `max_attempts = 3`
- retry выполняется только для retryable-ошибок
- между попытками применяется backoff

Пример backoff-стратегии для MVP:

```text
attempt 1 -> immediately
attempt 2 -> +5s
attempt 3 -> +30s
```

---

### 4. Поток выполнения на стороне расширения

Для каждого элемента очереди:

1. запросить следующий URL у localhost API
2. открыть или активировать вкладку с нужным видео
3. дождаться полной загрузки страницы
4. при необходимости инжектить `content.js`
5. вызвать `extractAndSaveMarkdown()`
6. определить успех или ошибку
7. отправить статус выполнения
8. перейти к следующему URL

Последовательная обработка выбрана осознанно: она уменьшает сложность, упрощает tracking статусов и снижает риск гонок при скачивании.

Состояния обработки фиксируются явно:

```text
queued -> in_progress -> done | error
```

При повторной попытке расширение должно отправлять тот же `job_id`, увеличивая `attempt`, а Python должен интерпретировать такие сообщения идемпотентно.

На стороне расширения должен существовать single-flight guard: в каждый момент времени только один job может находиться в активной обработке, даже если service worker был реактивирован или повторно инициировал polling.

Для каждого job должен быть задан `execution_timeout`, после которого обработка считается неуспешной и переводится в retry или `error` в зависимости от числа попыток. Для MVP допустим диапазон `60-120s`.

---

### 5. Изменения в manifest и permissions

Расширению понадобится явный доступ к локальному API, например:

- `http://127.0.0.1:8765/*`

Доступ должен быть максимально узким и ограничиваться конкретным портом.

---

### 6. Обработка ошибок

Ожидаемые классы ошибок:

- у видео отсутствует transcript
- YouTube изменил DOM, и extraction не сработал
- локальный API недоступен
- не удалось инициировать скачивание

Минимальная политика обработки:

- пометить текущий элемент как `error`
- продолжить выполнение со следующим URL
- вернуть машиночитаемое описание ошибки через `/report`

Дополнительные требования надежности:

- задания должны быть идемпотентными
- статус `in_progress` должен переживать перезапуск service worker
- рабочее состояние очереди на стороне расширения должно сохраняться в `chrome.storage`
- при истечении lease Python должен иметь возможность перевыдать задание
- reclaim должен быть явным: `lease expired -> job возвращается в queued`, `attempt += 1`
- retry должен быть ограниченным и явным
- retry должен использовать backoff, чтобы не создавать tight retry loop при постоянной ошибке
- после достижения `max_attempts` job переводится в финальный `error`

---

### 7. Надежность результата

Вызов `chrome.downloads.download` считается UX-операцией, но не достаточной гарантией успешного завершения batch-задачи.

Поэтому результат выполнения должен поддерживать два канала:

- `download` для привычного пользовательского сценария сохранения `.md`
- `report data` для надежной передачи результата обратно в Python API

Минимально в `/report` должны возвращаться:

- `job_id`
- `url`
- `status`
- `filename`
- `video_id`
- markdown-текст или другой структурированный payload результата

Это позволяет:

- не считать задачу успешной только по факту вызова Downloads API
- сохранять или валидировать результат на стороне Python
- в будущем легко добавить альтернативные стратегии сохранения

---

### 8. Состояние и восстановление

Из-за особенностей MV3 service worker архитектура должна предполагать потерю in-memory состояния.

Минимальные требования:

- активный `batch_id`
- активный `job_id`
- `attempt`
- временная метка lease
- текущий статус обработки

должны сохраняться в `chrome.storage` и восстанавливаться при реактивации service worker.

## Последствия

### Плюсы

- переиспользуется существующий pipeline извлечения в расширении
- не нужен Selenium, Playwright или контекстно-менюшная автоматизация
- решение работает с уже открытым Chrome
- модель эксплуатации проста и достаточна для небольшого суточного объема
- Python остается короткоживущим управляющим процессом
- reliability-слой делает решение пригодным не только для MVP, но и для устойчивого ежедневного использования

---

### Минусы

- в background/service worker появляется новый API-слой
- возникает координация между локальным Python-процессом и Chrome extension
- время жизни MV3 service worker нужно учитывать при проектировании polling/retry
- факт вызова `downloads.download` не полностью равен верификации содержимого файла на диске
- контракт и модель состояний становятся сложнее, чем в чистом MVP

---

## Рассмотренные альтернативы

### 1. Browser automation через Selenium или Playwright

Отклонено, потому что:

- для 20-30 роликов в сутки это избыточно
- добавляет хрупкость UI-автоматизации
- дублирует оркестрацию, которую расширение может выполнять само

---

### 2. Native Messaging

Отклонено на текущем этапе, потому что:

- сложнее в установке и регистрации
- тяжелее, чем требуется для локального single-user сценария
- localhost HTTP проще отлаживать и развивать итеративно

---

### 3. Прямое скачивание транскриптов в Python без расширения

Отклонено, потому что:

- дублируется логика, уже реализованная в расширении
- теряется преимущество текущей DOM-адаптации и auto-open transcript chain
- появляются два независимых способа решать одну и ту же бизнес-задачу

---

## Дальнейшее развитие

- заменить polling на push-модель при необходимости
- добавить выдачу structured transcript JSON помимо Markdown
- ввести batch id и поддержку resumable progress
- добавить на стороне расширения конфигурируемую стратегию сохранения результата

---

## Итог

Для небольшого ежедневного batch-сценария localhost API дает самый простой программный вход: Python координирует очередь, а расширение выполняет browser-native часть работы, переиспользуя существующий pipeline извлечения транскрипта.
